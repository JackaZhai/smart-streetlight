# 系统架构设计

## 1. 总体架构

```text
Browser
  |
  | HTTP 18080 / token login
  v
Frontend Nginx + Vue Static App
  |
  | /api /socket.io with Bearer token
  v
Backend Express + MQTT Bridge + Socket.IO
  |
  | RBAC / Audit / Fault Prediction / Local RAG / StateStore
  v
MySQL 8

Mock Device / 鸿蒙端
        |
        | MQTT telemetry / command reply
        v
      EMQX
        |
        v
Backend Express + MQTT Bridge + Socket.IO
```

## 2. 模块职责

| 模块 | 职责 |
| --- | --- |
| `mock-device` | 模拟光照传感器、设备心跳、路灯状态和控制回执 |
| `backend` | 接收 MQTT 数据、通过 `StateStore` 保存状态、判断阈值、生成告警、下发控制指令、故障预判、本地 RAG 问答、提供 REST API |
| `frontend` | Nginx 提供静态前端，并反代 API、健康检查和 Socket.IO |
| `deploy` | 提供 frontend、backend、EMQX、MySQL 全栈 Docker Compose 配置 |
| `mysql` | 持久化设备、阈值、光照历史、告警、控制日志和审计日志 |
| `knowledgeBase` | 本地运维知识库，支撑智能问答检索和引用片段 |

## 3. 数据流

1. 模拟设备定时向 `streetlight/{deviceId}/telemetry` 发布光照和心跳数据。
2. 后端 MQTT bridge 订阅 telemetry topic，通过 MySQL `StateStore` 更新持久化状态。
3. 后端根据阈值判断是否自动开灯或关灯。
4. 后端通过 `streetlight/{deviceId}/command` 下发控制指令。
5. 模拟设备执行指令并发布 `command/reply`。
6. 前端通过 Nginx 反代访问 REST API，首次加载数据后通过 Socket.IO 接收后续刷新。
7. 除健康检查和登录接口外，REST API 与 Socket.IO 均要求登录 token。
8. 后端基于 `admin/operator/viewer` 做角色授权，成功的关键写操作和被拒绝的越权操作写入审计日志。
9. 后端基于设备心跳、光照历史、控制日志和告警积压实时计算故障预判。
10. 智能问答先检索本地知识库，再结合设备上下文和故障预判生成答案、引用和推荐操作。

## 4. 后续替换点

- JSON 文件存储保留为本地开发兜底，产品化路径默认使用 MySQL 8。
- 本地 RAG 检索当前不依赖外部模型，后续可替换为 MaxKB、向量数据库或大模型问答服务。
- `mock-device` 可替换为鸿蒙端真实 MQTT 上报程序。
- 审计日志当前由 MySQL/JSON 状态层持久化，后续可扩展到独立日志库或 SIEM。
