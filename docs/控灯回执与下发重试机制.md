# 控灯回执与下发重试机制说明

## 1. 功能目标

控灯回执与下发重试机制用于补齐控制指令闭环，避免系统只记录“已下发”但不知道设备是否真正执行。

当前实现目标：

- 每条控制指令生成唯一 `commandId`。
- 后端下发 MQTT 指令时携带 `commandId`、当前尝试次数和最大尝试次数。
- 设备执行后通过 `command/reply` 回传 `commandId` 和执行结果。
- 后端收到回执后把对应控制日志从 `QUEUED` 更新为 `SUCCESS` 或 `FAILED`。
- 若指定时间内没有收到回执，后端自动重新下发。
- 默认最多下发 3 次，每次间隔 3 秒；超过次数仍未收到回执则标记控制日志为 `FAILED`。

## 2. 数据流

```text
前端点击开灯/关灯
  -> POST /api/devices/{id}/commands
  -> applyManualCommand() 生成 controlLog.id
  -> mqttBridge.publishCommand(commandId, ...)
  -> streetlight/{deviceId}/command
  -> mock-device 执行指令
  -> streetlight/{deviceId}/command/reply
  -> applyCommandReply()
  -> controlLogs.result = SUCCESS / FAILED
  -> Socket.IO state:update
```

自动控制链路同样复用该机制：

```text
MQTT telemetry
  -> applyTelemetry()
  -> evaluateAutomation()
  -> 生成 commandId 和 auto 控制日志
  -> MqttBridge 下发并等待 command/reply
```

## 3. 指令下发消息

Topic：

```text
streetlight/{deviceId}/command
```

Payload 示例：

```json
{
  "commandId": "control-xxxx",
  "deviceId": "SL-001",
  "command": "TURN_ON",
  "source": "manual",
  "attempt": 1,
  "maxAttempts": 3,
  "timestamp": "2026-07-03T03:01:00.000Z"
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `commandId` | 控制日志 ID，用于匹配回执 |
| `deviceId` | 设备编号 |
| `command` | `TURN_ON` 或 `TURN_OFF` |
| `source` | `manual` 或 `auto` |
| `reason` | 自动控制时可携带触发原因 |
| `attempt` | 当前下发次数 |
| `maxAttempts` | 最大下发次数 |
| `timestamp` | 本次下发时间 |

## 4. 控制回执消息

Topic：

```text
streetlight/{deviceId}/command/reply
```

Payload 示例：

```json
{
  "commandId": "control-xxxx",
  "deviceId": "SL-001",
  "command": "TURN_ON",
  "result": "SUCCESS",
  "timestamp": "2026-07-03T03:01:01.000Z"
}
```

后端优先使用 `commandId` 匹配控制日志。如果旧设备没有回传 `commandId`，后端会回退匹配同设备、同指令的最近一条 `QUEUED` 控制日志。

## 5. 重试机制

默认配置：

| 配置 | 默认值 | 说明 |
| --- | --- | --- |
| `retryDelayMs` | `3000` | 每次等待回执的时间 |
| `maxPublishAttempts` | `3` | 最大下发次数 |

处理规则：

1. 首次下发后记录为第 1 次尝试。
2. 在 `retryDelayMs` 内收到回执，则清理重试任务。
3. 收到 `SUCCESS` 回执，控制日志更新为 `SUCCESS`。
4. 收到 `FAILED` 回执，控制日志更新为 `FAILED`。
5. 未收到回执且尝试次数小于 `maxPublishAttempts`，使用相同 `commandId` 重新下发。
6. 达到最大次数仍未收到回执，控制日志更新为 `FAILED`。

## 7. 验收方式

本地测试：

```bash
npm --workspace backend test -- mqttBridge.test.ts streetlight.test.ts
```

全量验证：

```bash
npm test
npm run typecheck
npm run build
```

人工联调：

1. 启动 EMQX、后端、前端和 `mock-device`。
2. 使用 `admin / admin123` 登录。
3. 进入设备详情页点击开灯或关灯。
4. 确认 MQTT 指令 payload 含 `commandId`。
5. 确认 `mock-device` 发布 `command/reply`。
6. 确认控制记录由 `QUEUED` 更新为执行成功。
7. 停止 `mock-device` 后再次下发指令，确认后端最多重试 3 次并最终标记失败。
