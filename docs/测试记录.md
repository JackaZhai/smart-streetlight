## 人工联调流程

1. `cp .env.example .env`
2. `docker compose -f deploy/docker-compose.yml up -d`
3. `npm run dev:backend`
4. `npm run dev:frontend`
5. `npm run dev:mock`
6. 打开 `http://localhost:5173`
7. 检查实时光照、历史趋势、设备表格、告警列表和智能问答。

## 工程搭建与登录页验收点

1. 根目录 `package.json` 使用 npm workspace 管理 `frontend`、`backend`、`mock-device`。
2. `.env.example` 已覆盖 `PORT`、`FRONTEND_PORT`、`CORS_ORIGIN`、`MQTT_URL`、`STORAGE_DRIVER`、`DATABASE_URL`、`ADMIN_USERNAME`、`ADMIN_PASSWORD`、`JWT_SECRET` 和 `AUTH_USERS_JSON`。
3. `npm run dev:frontend`、`npm run dev:backend`、`npm run dev:mock` 分别对应前端、后端和模拟设备启动入口。
4. 登录页展示系统名称、账号输入、密码输入、演示账号说明和登录按钮。

8. 登录成功后前端保存 token，后续 API 请求携带 `Authorization: Bearer <token>`。
9. Socket.IO 实时连接使用登录 token 进行鉴权。

## MQTT 基础框架验收点

1. `deploy/docker-compose.yml` 使用 `emqx/emqx:5` 提供 MQTT Broker。
2. EMQX 对外开放 `1883` MQTT 端口和 `18083` Dashboard 端口。
3. 后端通过 `MQTT_URL` 创建 `MqttBridge` 并连接 Broker。
4. 后端订阅 `streetlight/+/telemetry` 和 `streetlight/+/command/reply`。
5. 模拟设备向 `streetlight/{deviceId}/telemetry` 定时发布光照、灯光状态、在线状态和时间戳。
6. 后端收到遥测后更新设备状态、写入光照历史，并触发自动控制判断。
7. 后端通过 `streetlight/{deviceId}/command` 发布开灯、关灯指令。
8. 模拟设备收到控制指令后更新本地灯光状态，并向 `streetlight/{deviceId}/command/reply` 发布执行回执。
9. 后端通过 Socket.IO 向前端推送 `state:update`，前端总览页可刷新设备状态和光照数据。
10. MQTT 主题格式和消息示例已记录在 `docs/MQTT主题设计.md`。

## MQTT 光照数据入库验收点

1. 模拟设备通过 `streetlight/{deviceId}/telemetry` 上报 `deviceId`、`lightIntensity`、`lampStatus`、`online` 和 `timestamp`。
2. 后端 `MqttBridge` 收到 `telemetry` 后调用 `applyTelemetry()`。
3. `applyTelemetry()` 更新设备在线状态、灯光状态、最后心跳时间，并追加 `LightReading`。
4. 光照历史按 `reportedAt` 排序，最多保留最近 240 条。
5. `StateStore.update()` 负责把更新后的状态写回当前存储。
6. JSON 本地模式下，光照读数可写入并从状态文件重新读取。
7. MySQL 产品化模式下，光照读数写入 `light_readings` 表。
8. `light_readings` 表包含 `device_id`、`light_intensity`、`lamp_status`、`reported_at` 等字段。
9. 后端提供 `/api/devices/{id}/latest-light` 和 `/api/devices/{id}/light-history` 查询最新光照与历史光照。
10. 前端总览和趋势图可基于入库后的光照历史刷新展示。

## 自动控灯与手动控灯验收点

1. 光照值低于 `lowThreshold` 且路灯当前为 `OFF` 时，后端自动生成 `TURN_ON` 指令。
2. 光照值高于 `highThreshold` 且路灯当前为 `ON` 时，后端自动生成 `TURN_OFF` 指令。
3. 自动控制指令写入 `controlLogs`，`source` 为 `auto`。
4. 阈值未启用、设备离线或设备关闭自动模式时，不触发自动控灯。
5. 手动控灯接口为 `POST /api/devices/{id}/commands`。
6. 手动控灯只允许 `admin` 和 `operator` 执行。
7. `viewer` 执行手动控灯会返回 HTTP `403`，并写入拒绝审计日志。
8. 手动控灯只接受 `TURN_ON` 和 `TURN_OFF`。
9. 手动控灯写入 `controlLogs`，`source` 为 `manual`。
10. 手动控灯写入 `auditLogs`，动作为 `command.dispatch`。
11. 自动控灯和手动控灯都会通过 `streetlight/{deviceId}/command` 下发 MQTT 指令。
12. 前端设备控制面板根据角色禁用或启用开灯、关灯按钮。

## 控灯回执与下发重试验收点

1. 每条自动或手动控制指令使用 `controlLog.id` 作为 `commandId`。
2. MQTT 控制指令 payload 携带 `commandId`、`attempt` 和 `maxAttempts`。
3. 模拟设备执行指令后通过 `streetlight/{deviceId}/command/reply` 回传 `commandId`。
4. 后端优先按 `commandId` 匹配控制日志，兼容旧设备按设备和指令回退匹配。
5. 收到 `SUCCESS` 回执后，控制日志 `result` 更新为 `SUCCESS`。
6. 收到 `FAILED` 回执后，控制日志 `result` 更新为 `FAILED`。
7. 未在 `retryDelayMs` 内收到回执时，后端使用相同 `commandId` 自动重试下发。
8. 默认最多下发 3 次，每次间隔 3 秒。
9. 达到最大尝试次数仍未收到回执时，控制日志 `result` 更新为 `FAILED`。
10. 回执成功、回执失败和最终失败都会触发 Socket.IO `state:update` 推送。

## 告警中心与设备台账验收点

1. 告警中心支持按级别、处理状态、设备关键字和日期范围筛选。
2. 告警统计展示未处理、中优先级、已处理和高优先级数量。
3. 告警详情展示处理人、处理时间和处理备注。
4. 告警处理后刷新总览、告警列表和审计日志。
5. 设备台账支持设备搜索、在线状态筛选和新增设备。
6. 设备列表展示在线状态、灯具状态、当前功率、今日能耗、最后心跳和故障预判。
7. 设备详情展示基础信息、光照趋势、控制记录、关联告警和故障预判。

## RAG 知识库与问答接口验收点

1. 后端本地知识库覆盖离线排查、光照异常、自动控灯、手动控灯、告警处理、阈值配置和 MQTT 主题。
2. 检索支持同义词归一化，如掉线/离线、控灯/控制、告警/故障。
3. 问题包含设备 ID 时，系统注入设备上下文。
4. `/api/agent/chat` 返回 `answer`、`provider`、`model`、`references`、`matches` 和 `suggestedActions`。
5. 配置 `DEEPSEEK_API_KEY` 后，问答底层调用 DeepSeek Chat Completions；未配置或请求失败时回退本地 RAG。
6. 前端智能问答展示模型来源、引用来源、知识命中片段和推荐操作。

## 故障预判与检索优化验收点

1. 后端基于设备心跳生成 `HEARTBEAT_TIMEOUT` 风险。
2. 后端基于光照极端值、连续不变或大幅波动生成 `LIGHT_SENSOR_ANOMALY` 风险。
3. 后端基于失败控制日志生成 `CONTROL_FAILURE` 风险。
4. 后端基于未处理告警积压生成 `ALARM_BACKLOG` 风险。
5. `/api/fault-predictions` 返回全部风险。
6. `/api/devices/{id}/fault-predictions` 返回单设备风险。
7. RAG 回答可将故障预判结果注入 DeepSeek 提示词并生成建议。
